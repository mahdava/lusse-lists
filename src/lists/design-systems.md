# Design systems to reference

| Name | URL | Stack | Why it matters | Priority | Notes |
|---|---|---|---|---|---|
| Helsinki Design System | https://hds.hel.fi/getting-started/ | React / Design system docs | Accessibility, civic design clarity, practical documentation | high | Strong local reference. Worth studying for docs clarity, tokens, and public-sector UX constraints. |
| Radix UI | https://www.radix-ui.com/ | React | Accessibility, primitives, behavior contracts | high | Very useful for interaction patterns, composition, and state/data-attribute conventions. |
| Headless UI | https://headlessui.com/ | React / Vue | Unstyled accessible primitives | high | Good behavior reference, especially when evaluating minimal abstraction. |
| Adobe Spectrum | https://spectrum.adobe.com/ | Cross-platform | Accessibility, design tokens, system rigor | high | Strong reference for design language governance and documentation structure. |
| Spectrum Web Components | https://opensource.adobe.com/spectrum-web-components/ | Web Components | Standards-based components, Spectrum implementation details | high | Useful when comparing React-first systems with framework-agnostic delivery. |
| Carbon Design System | https://carbondesignsystem.com/ | Cross-platform | Enterprise system thinking, documentation | high | Useful for content design, usage guidance, and pattern docs. |
| GOV.UK Design System | https://design-system.service.gov.uk/ | Web | Accessibility, content clarity | high | Extremely worth studying for forms and public-service UX. |
| U.S. Web Design System | https://designsystem.digital.gov/ | Government web | Accessibility, governance, content | high | High-value reference for public-facing product work. |
| Twilio Paste | https://paste.twilio.design/ | React | Excellent docs, strong component system | high | Very worth studying for docs structure and accessibility. |
| Primer | https://primer.style/ | React / CSS | Clear docs, practical product components | medium | Good benchmark for GitHub-style density and documentation clarity. |
| Polaris | https://polaris.shopify.com/ | React | Product UI patterns, guidelines, content clarity | medium | Good benchmark for UX writing and admin/product workflows. |
| Fluent 2 | https://fluent2.microsoft.design/ | Cross-platform | Large-scale system consistency | medium | Useful for token systems and design guidance at scale. |
| Material UI | https://mui.com/ | React | Broad component coverage, mature docs, theming | medium | Useful benchmark for component API surface and accessibility patterns. Avoid copying visual style blindly. |
| Chakra UI | https://www.chakra-ui.com/ | React | Consistent API design, theming, usability | medium | Good source for prop naming and ergonomic component contracts. |
| Mantine | https://mantine.dev/ | React | Wide surface area, hooks, strong docs | medium | Useful for app-level patterns and rich component behavior. |
| PatternFly | https://www.patternfly.org/ | React / Web | Enterprise admin UI | medium | Very worth tracking for tables, forms, and admin workflows. |
| shadcn/ui | https://ui.shadcn.com/ | React + Tailwind | Practical app patterns, copy-paste components, Radix-based | medium | Good reference for DX and Tailwind structure, but quality varies by component evolution. |
| Ant Design | https://ant.design/ | React | Enterprise patterns, dense component coverage | medium | Good for data-heavy UI references, less useful as a visual direction unless specifically desired. |
| Base UI | https://base-ui.com/ | React | Headless modern primitives | medium | Worth watching for API design and composability. |
| Nordhealth Design System | https://nordhealth.design/ | React / Design system docs | Clean product UI, healthcare-oriented patterns, consistency | medium | Worth reviewing for calm visual language and pragmatic product components. |
| Bulma | https://bulma.io/ | CSS framework | Simplicity, readable class model, low ceremony | low | Useful as a lightweight CSS reference, less useful for component behavior or accessibility contracts. |
| Apple Human Interface Guidelines | https://developer.apple.com/design/ | Platform design guidance | Platform-native interaction patterns, motion, clarity | low | Not a component library, but useful for interaction, hierarchy, and platform conventions. |
